# Workflow for a packed-bed parameter study (`Re = 0.001` to `100`) on `64^3`, `128^3`, and `256^3`

This document describes a practical workflow for running a Reynolds-number sweep for a packing generated by an external code. It is based on the current stabilized solver configuration in this repository and on the continuation utilities already present in `scripts/state_initialization.py` and `scripts/run_packing_multires_re_targets.py`.

## 1. Recommended overall strategy

For a new packing, use a **coarse-to-fine continuation workflow**:

1. generate or import the packing as an SDF field,
2. solve the sweep first on `64^3`,
3. interpolate each converged `64^3` state to initialize `128^3`,
4. interpolate each converged `128^3` state to initialize `256^3`,
5. compare the force needed to maintain the same target `Re` across the three grids.

This is the recommended approach because:

- `64^3` is cheap and useful for branch tracing,
- `128^3` is a good intermediate predictor,
- `256^3` is the production grid,
- direct low-Re-to-high-Re sweeps on `256^3` alone are much more expensive,
- interpolated lower-resolution states strongly reduce the transient settling on the finer grids.

## 2. Geometry generation and file format

### 2.1 What the solver expects

The solver expects a periodic signed-distance field (SDF) in VTI format.

Conventions:

- **SDF sign:** negative in solid, positive in fluid
- **domain:** periodic in all directions
- **storage:** structured scalar field read by `pnm.SDFReader.read_vti(...)` (the `pnm` module)

The solver is initialized with:

```python
import sdflow, pnm
# read_vti returns (sdf, origin_zyx, spacing_zyx)
sdf_zyx, origin, spacing = pnm.SDFReader.read_vti("data/packing_256.vti")
# sdflow runs in dimensionless grid units (unit spacing). set_solid wants a Fortran-order
# (nx, ny, nz) float64 SDF (negative inside solid); convert from the read order as needed
# (see scripts/verify_periodic_spheres_sdflow.py for the full pipeline).
solver = sdflow.Solver(nx, ny, nz)
solver.set_rho(1.0); solver.set_mu(1.0); solver.set_dt(dt)   # physical units; fix before geometry
solver.set_solid(sdf, cutcell_pressure=True, pressure_coarse="rediscretized")
```

### 2.2 Recommended way to create the three resolutions

If the external packing tool gives particle positions or a continuous geometry description, the best practical workflow is:

1. generate the **finest** SDF first (`256^3`),
2. use that same packing to derive `128^3` and `64^3`.

There are two ways to do that:

1. **Preferred if available:** voxelize / compute the SDF from the same packing separately at `64^3`, `128^3`, and `256^3` in the external tool.
2. **Simplest with current repo utilities:** generate `256^3` once and derive the coarser SDFs by periodic linear resampling.

The current repository already uses periodic linear resampling utilities in `scripts/state_initialization.py`:

- `resample_field_linear(field, target_shape)`
- `resample_state_linear(state, target_shape)`

The multi-resolution study script follows exactly this pattern.

### 2.3 Keep these geometric quantities consistent

Across `64^3`, `128^3`, and `256^3`, keep:

- the **same physical packing**,
- the **same domain size**,
- the **same origin**,
- nested spacing (`dx_64 = 4 dx_256`, `dx_128 = 2 dx_256`).

## 4. Reynolds number definition and scalar scaling

In the existing scripts, Reynolds number is defined as:

```python
Re_Do = u_mean * OUTER_DIAMETER
```

with

```python
OUTER_DIAMETER = 0.9498
```

This must match the characteristic length you want to use for your packing study. If your external packing has a different natural reference length, replace this constant consistently in the workflow.

### 4.1 Same-grid scalar continuation

For continuation on the **same resolution**, the repository already provides:

- `fit_force_law(u_mean_samples, force_samples)`
- `predict_u_mean(force, linear_coeff, quadratic_coeff)`
- `scale_solver_state(solver, previous_u_mean, target_u_mean, previous_force, target_force)`

in `scripts/state_initialization.py`.

The idea is:

1. from previous converged points, fit
   $$
   F \approx a u_{\text{mean}} + b u_{\text{mean}}^2
   $$
2. invert that relation to get a predicted target `u_mean`,
3. scale the current converged state with:
   - velocity factor `target_u_mean / previous_u_mean`
   - pressure/body-force factor `target_force / previous_force`

This is what `solver.scale_state(...)` is used for internally.

## 5. Cross-resolution interpolation

For transfer between resolutions, use:

- `extract_solver_state(solver)`
- `load_solver_state(solver, state)`
- `resample_state_linear(state, target_shape)`

from `scripts/state_initialization.py`.

For the multi-resolution script, the same logic is specialized as:

- `extract_state_3d(...)`
- `load_state_3d(...)`
- `resample_state_3d(...)`

in `scripts/run_packing_multires_re_targets.py`.

### 5.1 Recommended transfer path

At a fixed target `Re_k`:

1. solve `64^3` to convergence,
2. resample that converged state to `128^3`,
3. solve `128^3` to convergence,
4. resample that converged state to `256^3`,
5. solve `256^3` to convergence.

Do **not** use the interpolated state as the final answer. It is only an initial guess. Each grid must still be re-converged on its own operator.

## 6. Time-step strategy

The current stable continuation work uses:

```python
dt = 50.0 / force
```

This keeps `force * dt` roughly constant and has worked well up to `Re≈1000` in the stabilized mixed scheme.

### 6.1 Recommendation

Use this as the default pseudo-transient stepping rule for the sweep:

```python
dt = 50.0 / force
```

Then keep the following logic:

- run repeated time steps,
- every few steps, evaluate `u_mean`,
- declare stage convergence once the relative change in `u_mean` stays small for several checks.

The current script uses:

- sample every 5 steps,
- declare stable after 3 consecutive checks with
  $$
  \frac{|u_n-u_{n-1}|}{\max(|u_n|, 10^{-14})} < 10^{-3}
  $$

This is implemented in `run_to_convergence(...)` in `scripts/run_packing_multires_re_targets.py`.

## 7. How to choose the Reynolds-number sweep points

For `Re = 0.001` to `100`, use a **logarithmic continuation ladder**, not a uniform linear spacing.

Recommended example:

```python
TARGET_RES = [
    1e-3, 2e-3, 5e-3,
    1e-2, 2e-2, 5e-2,
    1e-1, 2e-1, 5e-1,
    1, 2, 5,
    10, 20, 40, 60, 80, 100,
]
```

If the packing shows strong nonlinearity or solver difficulty in a subrange, insert extra points there, for example:

- `Re = 15, 30, 50, 70, 90`

The dense low-Re section is important because:

- it seeds the force law cleanly,
- it gives robust continuation startup,
- it reduces bad force extrapolations later.

## 8. How the target-Re solve should work

For each target `Re_k` on each grid:

1. set an initial body-force guess,
2. choose `dt = 50 / force`,
3. run to pseudo-transient convergence,
4. compute achieved `Re`,
5. update force by interpolation/extrapolation in the `(force, Re)` relation,
6. scale the converged state to the updated force,
7. repeat until the achieved `Re` is within tolerance.

The current implementation does exactly this in:

- `solve_target_re(...)`
- `update_force_guess(...)`

inside `scripts/run_packing_multires_re_targets.py`.

The current target-Re tolerance is:

```python
TARGET_RE_TOL = 0.015
```

That is a relative tolerance of about `1.5%`. For a production study you may want to tighten this somewhat, especially at the higher resolutions.

## 9. Suggested workflow for a new packing

### Step 1: generate the SDF

Create at least:

- `packing_256.vti`

Then either:

- create `packing_128.vti` and `packing_64.vti` externally too, or
- derive them from `packing_256.vti` by periodic resampling.

### Step 2: adapt the multi-resolution script

Use `scripts/run_packing_multires_re_targets.py` as the template.

Adapt:

- `TARGET_RES`
- `OUTER_DIAMETER`
- `OUTPUT_DIR`
- input file names in `build_sdf_catalog()`

If you already have explicit `64^3` and `128^3` VTI files, replace the current resampling step in `build_sdf_catalog()` with direct reads.

### Step 3: run the sweep

Run the coarse-to-fine study:

```bash
python3 scripts/run_packing_multires_re_targets.py
```

This should:

- solve all target `Re` points,
- save one converged state per grid and per target,
- write summary CSV files,
- generate diagnostic plots.

### Step 4: inspect convergence with resolution

Compare, at fixed target `Re`:

- required force,
- number of pseudo-transient steps,
- `outer_total`,
- optionally pressure drop or drag-derived observables if you add them.

The current script already plots:

- force vs `Re`
- outer_total vs `Re`
- steps vs `Re`
- relative force error vs `256^3`

## 10. Storage considerations

Saving every converged state is useful, but `256^3` states are large.

In the current study:

- `64^3` state: a few MB
- `128^3` state: a few 10s of MB
- `256^3` state: a few 100s of MB

So for many Reynolds points, state storage can become large quickly.

Practical options:

1. **Save all `64^3` and `128^3` states**
2. **Save all `256^3` states only if storage is acceptable**
3. Otherwise save only:
   - selected milestone points,
   - failed/problematic points,
   - the final converged `256^3` states needed for post-processing

## 12. Practical recommendation

For a new packing from an external code, the most robust production workflow is:

1. generate the finest packing SDF (`256^3`),
2. derive or generate matching `128^3` and `64^3` SDFs,
3. run a logarithmic target-`Re` sweep from `0.001` to `100`,
4. use `64^3 -> 128^3 -> 256^3` cross-resolution state interpolation at each target `Re`,
5. use `dt = 50 / force`,
6. use the current stabilized mixed scheme and the solver settings from `build_solver(...)`,
7. analyze convergence by comparing the force required to maintain the same `Re`.

If the `128^3 -> 256^3` shift remains large at the top of the range, treat `256^3` as the quantitative result and `64^3`/`128^3` primarily as accelerators for continuation.
